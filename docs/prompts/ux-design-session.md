Ты проектируешь UX для **Obrato** — web-PWA помощника прораба (ремонт, Лиссабон).
Из замеров комнат + состава работ приложение детерминированно считает количества
материалов по нормам, собирает список закупок по этапам стройки и редактируемые
2D-чертежи. **Это инструмент прораба** (не дизайнера, не заказчика). Пилот N=1 —
прораб Давид, на объекте, спешит, не технарь, телефон Android (возможно в перчатках).

## Задача
1. **Потоки и экраны** — спроектировать пользовательские флоу.
2. **Модель ввода** — закрыть **Open Q4**: минимальный набор галочек «состав работ», покрывающий ~80% объектов Давида.
3. **Визуальный дизайн** ключевых экранов — через скилл **`frontend-design`** (production-grade UI; или gstack `design-shotgun`/`design-consultation` для генерации и сравнения вариантов). Стек — React + Canvas.

## Экраны (из гл.09)
ProjectList → ProjectDetail → RoomEditor (размеры Д×Ш×В, работы галочками, wet, проёмы; шаблон по типу комнаты предзаполняет → правка дельты) → LayoutCanvas (выбор плитки, редактируемая раскладка, tap-to-cut/suppress/annotate) → PurchaseList (по этапам, количества-диапазоны + цена-«ориентир», freeze «взять в магазин») → ProcurementEntry (факт закупки, часто офлайн в магазине).

## Ключевые UX-вызовы
- **Минимум ввода** (П3): прораб в операционке. Шаблон предзаполняет, прораб правит только дельту. Какой минимум галочек реально нужен? (Q4)
- **Честная неопределённость** (D7, гл.07): диапазон + «уточни по факту/первому слою», не ложно-точная цифра. ДВА уровня confidence (норма vs количество в листе — см. гл.09 §1, не путать).
- **Редактируемый чертёж** (D8): наивная раскладка плитки + ручная правка датума/подрезки тапом по Canvas. Pilot — только `pattern=grid`.
- **Офлайн на стройке**: крупные таргеты под палец/перчатку, всё работает без сети, ненавязчивый индикатор синка (OfflineBadge).
- Минимум текста, максимум «нажал — получил лист».
