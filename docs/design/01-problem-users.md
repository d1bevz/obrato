# 01 · Проблема, пользователи, посылки

## Problem Statement

Прорабы и бригадиры на ремонте/стройке теряют деньги из-за нехватки нужных
материалов в нужный момент. Корень — короткий горизонт планирования: прораб
весь в операционке, планировать закупки наперёд некогда, а по ходу работ
реальность постоянно «едет» (перерасход, переделки, забытый дозаказ).

Идея — приложение **Obrato** (web-PWA, AI-ассистент прораба): из замеров помещения и
состава работ оно детерминированно строит технические чертежи (план полов,
развёртки плитки, электрика), считает количества материалов по нормам расхода,
позволяет выбрать конкретные материалы и собирает локализованный список закупок
(Leroy Merlin и локальные магазины Лиссабона) с ценами, наличием и разбивкой по
этапам стройки, плюс напоминания о своевременном дозаказе.

Практический wedge: не «управление стройкой вообще», а **закупочный планировщик
для прораба** — замеры → количества → округление до упаковок → локальные SKU →
закупка по этапам.

**Это инструмент прораба, а не дизайнера и не конечного заказчика.** Эстетику/
дизайн приложение НЕ генерирует — оно переводит замеры и решения прораба в
точную смету и план закупок.

## Demand Evidence (Q1)

- Друг Давид (прораб, стройка/ремонт, Лиссабон) **теряет деньги** из-за нехватки
  материалов и **постоянно психует, что такого инструмента нет** — носит боль с
  собой, активно ищет решение.
- Конкретный механизм потери: нет материала → едет по магазинам / простой →
  теряет день → **не может взять следующий заказ** → каскадный срыв цепочки.
- Озвученная цифра: **~1000 €/день простоя.**
- Давид **готов платить**.

> Честная оговорка: это пока **N=1**. «Друг готов платить» и «постоянно психует» —
> сильнее, чем waitlist-интерес, но это ещё не доказанный спрос рынка. Доказательство —
> Давид, который реально пользуется и платит (см. Assignment).

## Status Quo (Q2)

Сегодня Давид решает проблему «на руках»: короткий горизонт планирования (он в
операционке), материалы заказываются реактивно, по ходу что-то кончается/меняется,
дозаказ происходит с опозданием → простой. Никакого систематического инструмента
планирования закупок под объект у него нет.

Природа боли двойная и подтверждена Димой: **и провал планирования** (не успел
подумать наперёд), **и провал исполнения** (по ходу поехало). Поэтому продукт
обязан и планировать наперёд, и ловить отклонения в процессе.

## Target User & Narrowest Wedge (Q3 + Q4)

**Пользователь и плательщик:** прораб / бригадир (Давид и подобные). Не дизайнер,
не конечный заказчик квартиры.

**Narrowest wedge:** замеры помещения + состав работ галочками → количества
материалов по нормам → выбор конкретики → **локализованный список закупок по
этапам** (Leroy PT / локально: цена, наличие, аналоги) + пинг на дозаказ.

Ключевая переоценка, случившаяся в ходе сессии: материалы **не нужно вычислять
из фотографий или восстанавливать computer vision-ом**. Количества выводятся
геометрически из введённых размеров + состава работ + норм расхода. Самый
дорогой и рискованный кусок, который изначально казался ядром (CV, лазерная
реконструкция, AI-рендеры), оказался не нужен вовсе.

## Premises

- **П1 — подтверждено.** Пользователь и плательщик = прораб. Не заказчик, не дизайнер.
  Продукт = AI-ассистент прораба.
- **П2 — подтверждено.** Боль = нет материала в момент → простой → срыв следующего
  заказа → ~1000 €/день. Дорого, повторяемо, остро ощущается.
- **П3 — подтверждено.** Корень = нет времени/горизонта на планирование (операционка).
  Следствие: продукт обязан требовать минимум ввода и планировать за прораба.
  Инструмент, требующий много ручного планирования, не приживётся.
- **П4 — подтверждено.** Не генерируем эстетику. Вход = замеры (ввод чисел) + состав
  работ галочками по комнате. Выход = чертежи + количества → выбор → закупки.
- **П5 — уточнено.** Главный *рыночный* ров = **слой локализации**: количества →
  список закупок Leroy PT/локально (наличие, цена, аналоги) + разбивка по этапам +
  пинг на дозаказ. НО: **точность норм расхода — это не коммодити, а gating-допущение
  и риск №1** (см. «Главные риски»). Локализация — ров; нормы — то, без чего ров
  бесполезен. Эти две вещи в первой версии дока были перепутаны.
- **П6 — пересмотрено.** Визуализация в MVP = **детерминированные 2D-чертежи из
  размеров** (планы, развёртки плитки, электрика), НЕ AI-рендеры и НЕ image-модели.
  Это убирает риск «обещал-не сделал» и сохраняет точность.
