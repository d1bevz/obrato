# 02 · Продукт: подход, scope, дистрибуция

## Approaches Considered

> Платформа в ходе ревью пересмотрена: «нативный Android» → **web-PWA** (D1, гл.03/04).
> Подходы ниже — историческая фиксация office-hours; «мобильное / Android-приложение»
> читать как «полноценное приложение Obrato».

### Approach A: Telegram-бот за выходные (минимальный)
Бот считает количества + текстовый список закупок. **Отклонён** Димой — нужен
визуальный 2D-план (чертежи), который в боте не живёт; нужна полноценная аппка,
не бот.
- Effort: S | Risk: Low
- Reuses: Euler-бот инфра, Grok API, scheduled-tasks
- Минус: не покрывает чертежи-картинки, не та платформа

### Approach B: Мобильное приложение с геометрическим движком (выбрано)
Приложение: ввод размеров + состав работ → детерминированный движок строит
2D-чертежи (план полов, развёртки плитки, электрика) + считает количества по нормам
→ выбор материалов → локализованный список закупок (Leroy PT) по этапам + пинги.
- Effort: L | Risk: Med
- Reuses: TypeScript/Claude-Code паттерны и Postgres-опыт из Mealset; gstack browse
  для сбора данных Leroy
- Плюс: цельное видение, точность, ров в каталоге-движке, путь к B2B
- Минус: геометрический движок + раскладки — не «выходные»; дольше до первой ценности

### Approach C: Concierge без кода (латеральный)
Давид кидает замеры+пожелания → список закупок собирается руками → отдаётся. 2-3
объекта для выяснения реальных норм/материалов/формата, потом код.
- Effort: S | Risk: Low
- Плюс: ноль кода, максимум обучения, не угадываешь нормы
- Минус: не масштабируется, не «продукт» (Дима хочет строить систему)

## Recommended Approach

**Approach B (приложение с геометрическим движком — платформа уточнена до web-PWA,
D1), но первые 2-3 реальных объекта вести в режиме Approach C** (Давид получает закупочный лист, а Дима/Давид руками сверяют каждый
расчёт количеств и каждую позицию закупок).

Обоснование:
- B соответствует цельному видению Димы и выбранной платформе (web-PWA, D1).
- C-режим на первых объектах = «тест в бою»: ловит, где будущий движок соврал бы
  по количествам и где локальный каталог неполон, до того как прорабу доверят
  расчёт вслепую.
- Не строить «идеальную систему» на догадках о нормах — узкий MVP (прямоугольные
  комнаты, топ-материалы по слоям, один Давид), расширение после подтверждения.

**MVP scope (предложение):**
1. Ввод объекта: помещения, размеры (Д×Ш×В), проёмы.
2. Состав работ галочками по комнате: пол / стены / потолок / **электрика-пакет**
   (электрика-пакет MVP = розетки, выключатели, точки света, щит — только
   **позиции точек**, без расчёта трасс кабеля; количество точек = по шаблону на тип
   комнаты + ручная правка).
3. **2D-план полов** (тривиально) + **развёртка плитки «мокрых» зон**. Важно про
   плитку: MVP = **наивная центрированная сетка, явно помеченная «черновая
   раскладка»**. Балансировка подрезки по правилам тайлера (датум, обход
   проёмов/сантехники, паттерны) — это отдельный solver, **фаза 2**, не «начертить
   сетку» (см. «Главные риски»).
4. Расчёт количеств по нормам расхода (площади + коэффициенты запаса/подрезки) →
   **округление до товарной упаковки** (плитка — коробки по N м², краска — банки,
   клей — мешки, профиль/плинтус — хлысты). Округление взаимодействует с
   коэффициентом отхода — это часть ядра, не «потом».
5. Выбор материалов из курируемого каталога локальных аналогов (Leroy PT).
6. Список закупок **по этапам стройки** (канонический порядок этапов — см. ниже) +
   цена + ссылка/наличие + пинг на дозаказ.

Сознательно ВНЕ MVP: AI-рендеры, лазерная Bluetooth-интеграция, **solver
балансировки подрезки плитки**, разводка электрики трассами, непрямоугольная
геометрия, мультипользовательский/командный режим, **активное отслеживание
отклонений по ходу** (drift) — в MVP только upfront-планирование + ручной дозаказ;
ловля отклонений в процессе = фаза 2 (см. риск про двойную природу боли).

**Канонический порядок этапов** (драйвит и группировку закупок, и тайминг пингов):
демонтаж → черновая сантехника + электрика → стяжка/выравнивание пола →
гидроизоляция мокрых зон → плиточные работы → отделка стен/потолков (краска) →
напольные покрытия → чистовая: сантехника, электроустановка, двери, мебель.

## Success Criteria

- **Ближний (пилот, N=1):** Давид использует пилотный закупочный лист на ≥3
  реальных объектах; список закупок он реально несёт в магазин; зафиксировано
  снижение/исчезновение простоев из-за нехватки материалов; Давид платит.
- **Качество движка:** на сверке (C-режим) количества материалов сходятся с
  фактической закупкой в пределах разумного коэффициента запаса.
- **Средний:** второй прораб (не Давид) проходит онбординг и доводит объект до
  списка закупок без ручной помощи Димы.

## Distribution Plan

- **Пилот (N=1):** web-PWA по прямой ссылке (Давид ставит на домашний экран), без стора.
- **Дальше:** Google Play через TWA-обёртку поверх PWA (D1). Для B2B-вектора — прямые
  продажи стройкомпаниям через контакты Димы (вектор, не MVP).
- CI/CD и деплой PWA — настроить, когда выйдем за пределы ручной раздачи (гл.09 §5).

## The Assignment

**Возьми ОДИН реальный объект Давида на этой неделе и пройди его в режиме concierge —
без единой строчки кода.** Давид даёт замеры и состав работ; ты руками собираешь
список закупок по Leroy PT с количествами, ценами и разбивкой по этапам; Давид
несёт его в магазин и на объект.

Цель — не продукт, а данные: какие материалы повторяются, какие нормы расхода
реальны, где Leroy не хватает (нужны локальные магазины/аналоги), в каком формате
лист реально используется на объекте, и — главное — **сэкономил ли это Давиду
простой**. Это и есть фундамент, на котором потом строится приложение, и живой
кейс для питча стройкомпаниям.
