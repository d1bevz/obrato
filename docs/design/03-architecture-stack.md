# 03 · Архитектура и стек

## Constraints

- **Платформа: web-PWA, таргетящая Android** (пересмотрено 2026-05-29; ранее
  «нативный Android-app»). Android = целевой рантайм/полигон, не среда сборки.
  Прогрессивное веб-приложение: ставится на домашний экран, работает офлайн
  (service worker + IndexedDB — критично на объекте без сети), ощущается как
  нативное. Не Telegram-бот. Нативку (Flutter/Kotlin) отвергли: для нагрузки
  «формы + 2D-вектор + лёгкий счёт» у веба нет потолка плавности, а нативный
  стек — не профиль соло-DS и худший ROI. Google Play позже — TWA-обёртка
  поверх PWA. Детали стека и архитектуры — раздел «Стек и порядок сборки».
- **Замер:** ручной ввод чисел с лазерного дальномера (напр. Bosch GLM 40) — прибор
  уже есть у строителя, замер и так этап любой стройки. Bluetooth-интеграция с
  дальномером — возможная фаза 2, не обязательна для MVP.
- **Геолокация ценности:** Лиссабон и окрестности, Leroy Merlin Portugal + локальные
  магазины.
- **Языки:** интерфейс — как минимум PT (Давид/рынок), вероятно + RU/EN.
- **Визуализация — детерминированная**, из чисел (SVG/Canvas-чертежи), без ML-генерации.

## Стек и порядок сборки (решено 2026-05-29)

**Стек.** web-PWA. UI — React + Vite (vite-plugin-pwa). Плотная графика (раскладка плитки) — **Canvas** (immediate-mode), НЕ
тысячи SVG-нод; планы полов и маркеры электрики — можно SVG. Тяжёлый счёт (нормы,
будущий solver подрезки) — в **Web Worker**, вне UI-потока (причина залипаний №1 —
расчёт на главном потоке). Офлайн — service worker + IndexedDB. **Язык ядра — Rust → WASM**
(решение 2026-05-29, см. «Решения» D3); UI — TypeScript/React; граница —
wasm-bindgen + serde, TS-типы генерятся из Rust (tsify/ts-rs). Профайлинг — на реальном телефоне класса Давида, не на
десктоп-Chrome и не на флагмане.

**Архитектурный инвариант: `compute-core` — отдельный чистый модуль.** Чистые
функции, юнит-тесты, ноль зависимостей от UI/DOM. Поток: модель ввода → таблица норм
(число + допущения) → количества → округление до упаковок → геометрия-данные (Canvas рисует, не SVG-строки) → лист
закупок по этапам. Тот же модуль крутится и в PWA (клиентский расчёт), и headless для
генерации PDF на пилоте. Когда лист соврёт в количествах — баг пинится в тестах
движка за минуту, а не ищется по всему быстро-сгенерённому аппу.

**Порядок и метрика скорости.** Строим сразу полноценно с AI, итерируем быстро. НО:
AI ускоряет код, не ускоряет проверку норм — она идёт со скоростью реальных строек, не
промптинга. Поэтому метрика прогресса = **как быстро `compute-core` встретился с
фактической закупкой Давида на живом объекте**, а не как быстро отполирован UI. Первый
прогон — один реальный объект Давида, выход = PDF-лист в магазин, ручная сверка каждого
числа с реальной закупкой. Это и есть валидация Риска №1, и она не делегируется AI.
