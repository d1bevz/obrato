# 11 · План пилота / roadmap

> Этап планирования. Здесь — последовательность сборки, вехи и метрики. Не код, не сроки в днях (они вторичны к скорости реальных строек Давида).

## 11.1 Цель пилота и метрика успеха

**Цель пилота (N=1, Давид, Лиссабон):** доказать, что compute-core выдаёт количества материалов, которым прораб доверяет настолько, что закупает по ним — и факт сходится с оценкой в пределах заранее объявленного допуска.

Всё остальное (UI, чертежи, offline, sync) — обвязка вокруг этой проверки. Красивый UI на неоткалиброванных нормах = демо, а не продукт.

**Метрика успеха = D5: точность норм на реальном объекте.**

Единица данных (фиксируем явно, иначе N плывёт): **точка данных = (материал × поверхность/комната)**. «Объект» = набор точек. Критерии формулируем в точках данных по материалу, а не в «объектах» — для N=1 с медленным потоком это разница между «выборка наберётся за недели» и «никогда».

Для каждой точки фиксируем тройку (инвариант гл.05): `estimate` (что выдало ядро) → `actual` (что реально ушло) → `reconciliation` (почему разошлось). Рядом пишем `david_gut` — оценку прораба «на глаз» до закупки: ядро обязано быть не хуже его интуиции, иначе оно не нужно.

**Пред-регистрируем числа ДО первого слайса** (гипотеза, данные уточнят — но записаны заранее, чтобы пилот можно было провалить):

- **Допуск прохождения:** `in_range` = `actual` внутри границ диапазона нормы гл.07. Если диапазон уже ±15%, прохождение = `|delta%| ≤ 15%` по позиции. Допуск шире ±50% не считается прохождением (тривиален и бесполезен).
- **Beats-baseline:** оценка ядра не хуже `david_gut` (ближе к `actual` или равна). Ядро, проигрывающее интуиции прораба, пилот не проходит, даже если формально «в диапазоне».
- **Правило калибровки:** ≥ 3 наблюдения одного знака подряд с `delta` вне диапазона по материалу → норму двигать (не «ошибка пользователя»); единичный выброс с понятной причиной → reconciliation, норму не трогаем.
- **Агрегат пилота:** для ≥ 4 из 5 ключевых материалов (клей, затирка, гидро, краска, плитка) факт стабильно `in_range` ИЛИ норма откалибрована и следующая точка попала.

Анти-метрика (за чем НЕ гонимся): скорость UI, число фич, гладкость онбординга, multi-device. Метрика прогресса — **как быстро ядро встретилось с реальной закупкой Давида**, а не как отполирован экран.

## 11.2 Build-slice: тонкий вертикальный слайс вверх

Принцип: каждый слайс приближает момент «посчитали → Давид купил → сверили факт». Порядок выбран так, чтобы Risk #1 проверялся как можно раньше и дешевле. Ключевой разворот относительно интуиции: **первым Risk #1 де-рискует не код, а карандаш.**

**Слайс −1 — Бумажная валидация (де-рискует Risk #1 сильнее всего, ноль кода).**
Взять 1–2 уже завершённые стройки Давида. Посчитать seed-нормы гл.07 вручную (Excel/калькулятор) по их замерам. Сверить с тем, что он реально закупил. Это даёт первый сигнал по Risk #1 за дни — без Rust, WASM и единого пикселя. Если нормы врут уже здесь — это узнаётся до настройки тулчейна, а не после.
*Зачем самым первым:* проверка нормы не требует ни строки кода. AI ускоряет код, но не ускоряет эту сверку — значит, её нельзя ставить за кодом.

**Слайс 0 — Ядро вслепую (автоматизация доказанной арифметики).**
Rust compute-core как чистый модуль + нативные `cargo test`. Вход: замеры + список работ. Выход: количества по seed-нормам с диапазоном и confidence. Ядро держится **примитивным**: чистая арифметика по нормам, ноль зависимостей сверх std, без solver (отложен, гл.09). Никакого UI/WASM/браузера. Прогон воспроизводит уже проверенную вручную (слайс −1) сверку — ядро здесь автоматизирует доказанную арифметику, а не служит местом первой проверки нормы.
*Зачем вторым:* норму уже проверил карандаш; задача слайса — закрепить её в тестируемом коде и заодно безопасно встретить Rust-тулчейн на примитивной задаче.

**Слайс 1 — Числа на бумагу (первый сквозной артефакт).**
Минимальный путь ядро → PDF-лист закупок по этапам. **PDF генерируется из нативного Rust-бинаря** (`cargo run` → PDF/CSV) — WASM/браузер НЕ требуются для валидации нормы. Ввод замеров грубый (форма/таблица, без Canvas). Лист **округляется до реальных фасовок** магазинов Давида (мешок клея = X кг, ведро краски = Y л), иначе им нельзя пользоваться в магазине. Цель: Давид получает реальный лист, по которому идёт в магазин.
*Зачем третьим:* PDF — продукт для проверки нормы в поле; ценен даже без чертежей.

**Слайс 2 — Сверка-петля (замыкает цикл калибровки).**
Форма ввода факта: по каждой точке `actual` + `reconciliation` (фиксированный enum, см. 11.4). Хранение тройки с дня 1 (инвариант гл.05). Отчёт «оценка vs факт» с **разделением `norm_delta` и `packaging_delta`** — иначе округление до фасовки загрязнит сигнал точности нормы.
*Зачем четвёртым:* без записи факта пилот не измеряем. Это инструмент самой метрики D5.

**Слайс 3 — Чертёж и Canvas (продуктовая ценность поверх доказанной нормы).**
Императивный Canvas: редактируемые 2D-чертежи, замеры с чертежа кормят ядро. Здесь же оправдан полный путь Rust→WASM→Web Worker — потому что Давиду нужно вводить данные на объекте. Только после того как нормы начали сходиться, иначе полируем фантик.

**Слайс 4 — Offline-first и обвязка платформы.**
Service worker + IndexedDB, outbox, server-assigned revision, sync (D-блоки). Когда есть что синхронизировать и доказанная ценность, ради которой Давид носит приложение на объект.

> Слайсы −1…2 — это и есть «build-slice»: ядро встречается с реальной закупкой Давида ДО полировки UI, а первый сигнал по норме приходит ещё раньше — из бумаги. Слайсы 3–4 наращивают продукт поверх уже доверенной нормы.

## 11.3 Вехи (milestones)

Вехи привязаны к валидации норм, а не к фичам. «Готово» = что-то проверено на реальной закупке.

- **M0-prep — Карандаш сошёлся.** По 1–2 прошлым стройкам Давида seed-нормы посчитаны вручную и сверены с реальной закупкой; видно, где норма попала. Первый сигнал по Risk #1 без кода. *(слайс −1, блокер M0)*
- **M0 — Ядро считает.** `cargo test` зелёный; ядро воспроизводит бумажную сверку M0-prep в пределах диапазона. *(слайс 0)*
- **M1 — Первый реальный лист.** Давид получил PDF-лист (округлённый до фасовок) по реальной комнате и сходил с ним в магазин. *(слайс 1)*
- **M2 — Первая сверка.** По объекту записаны estimate/actual/reconciliation с разделением norm/packaging-delta; видим, где норма попала, где промахнулась. *(слайс 2)*
- **M3 — Тренд по точкам.** Набрано ≥ 3 независимых точки данных по каждому ключевому материалу; по каждой seed-норме принято решение: подтвердить / сдвинуть / нужно ещё данных (детерминированно по правилу калибровки из 11.1). *(критерий пилота)*
- **M4 — Доверие (измеримо).** На объекте Давид закупился так, что число купленного = числу из листа (а не пересчитанному вручную), подтверждено чеком/фото; ручной override строк листа < заранее заданного X%. Засчитываем «купил ровно по листу», а не «сказал что доверяет». *(сигнал, что compute-core прошёл)*
- **M5 — Продукт поверх нормы.** Чертёж/Canvas + offline включены, потому что норма заслужила, чтобы вокруг неё строили продукт. *(слайсы 3–4)*

M0-prep→M4 — ядро пилота. M5 — переход от «доказать норму» к «сделать удобно».

## 11.4 Что валидируем на объекте и как

**Что:** на каждой точке (материал × поверхность) — расход ключевых материалов против seed-норм гл.07 (клей 3.5, затирка 0.3, гидро 3.5 kg/m², краска 0.17 L/m², плитка с подрезкой и т.д.).

**Как (формат сверки):**

| поле | смысл |
|---|---|
| `estimate` | количество от ядра (диапазон + confidence), **до округления до фасовки** |
| `david_gut` | оценка прораба «на глаз» до закупки (baseline-to-beat) |
| `actual` | сколько реально ушло (купленные фасовки пересчитаны обратно в кг/л) |
| `delta` | расхождение `actual` vs `estimate` (абс. + %) |
| `in_range?` | факт внутри диапазона нормы (или `\|delta%\| ≤ допуск`) |
| `beats_gut?` | оценка ядра не хуже `david_gut` |
| `norm_delta` / `packaging_delta` | расхождение по норме vs артефакт округления до фасовки |
| `reconciliation` | фиксированный enum (ниже) |

**Reconciliation — фиксированный enum** (свободный текст не агрегируется в решение):

- `cutting` (подрезка), `waste` (бой) — при ожидаемом уровне считаются **сигналом нормы** (норма должна их учитывать);
- `buffer` (запас прораба), `reorder` (перезаказ), `one_off` (разовый выброс) — **шум**, норму не трогают.

**Как читаем результат:**
- сверка идёт на уровне **расчётного количества** (до округления до фасовки); «купил X мешков» пересчитывается обратно в кг/л перед сравнением с `estimate`;
- факт в диапазоне и не хуже gut → норма подтверждается (растим confidence);
- ≥ 3 наблюдения одного знака с norm_delta вне диапазона → кандидат на сдвиг нормы (ручная калибровка в пилоте; авто-калибровка отложена, гл.09);
- M3-вердикт по каждой норме (подтвердить/сдвинуть/нужно данных) — детерминированный read по enum, а не суждение «на глаз».

## 11.5 Параллельные не-build треки

Идут одновременно со сборкой, скоростью реальных строек не лимитированы. **Не путать с критическим путём:** сбор прошлых закупок и фасовки — НЕ здесь, они подняты в слайсы −1/1 как блокеры (см. 11.2).

- **Brand-homework (из office-hours):** занять домен `obrato.com` / `.app` / `.pt`; проверить имя «Obrato» в Google Play; занять `@obrato` в соцсетях. TM — позже, пилот не блокирует.
- **Состав работ от Давида:** реальный список типовых работ и их последовательность по этапам (вход для группировки листа закупок по этапам).
- **Расширение потока данных (митигация N=1):** собрать НЕСКОЛЬКО прошлых строек Давида сразу (засеять несколько материалов параллельно); опционально — второй дружественный прораб как **чистый источник данных** (не пользователь продукта, scope не растёт), чтобы ускорить валидацию норм. Цель — N точек данных на материал, а не N продуктовых юзеров.

## 11.6 Явно вне пилота

Зафиксировано как deferred в гл.05/09 — не строим на N=1:

- solver автоматической раскладки плитки;
- multi-device / совместная работа нескольких устройств;
- авто-калибровка норм (в пилоте калибруем вручную по reconciliation);
- B2B-фичи (команды, роли, биллинг сверх нужд одного прораба);
- интеграция Bluetooth-дальномера;
- всё, что требует > 1 пользователя, чтобы иметь смысл.

Каталог материалов — глобальный read-only; нормы — диапазон + флаг калибровки (инварианты стека).

## 11.7 Риски графика и как их не словить

- **Yak-shaving тулчейна (главный риск темпа).** Rust→WASM→Web Worker→service worker — соблазн вылизать пайплайн до первого реального объекта. Противоядие тройное: (1) первый сигнал по норме приходит из **бумаги** (слайс −1), не из кода; (2) слайс 0 — нативный `cargo test`, без WASM; (3) первый PDF (слайс 1) — из нативного Rust-бинаря, WASM не предусловие валидации. WASM появляется не раньше слайса 3 (ввод на объекте). Ловлю себя на настройке тулчейна, пока реальный объект не приближается — стоп.
- **Rust как обучение на критическом пути.** Rust для Димы нов. Развязка: норму снимает бумага (слайс −1) без Rust; первая Rust-реализация держится примитивной (арифметика, нативный тест, ноль зависимостей) — это и де-риск тулчейна, и обучение, но НЕ блокер первого численного сигнала.
- **Полировка UI впереди нормы.** Canvas красив и приятно делается — отложен до слайса 3, после первых сходящихся сверок. Метрика — не гладкость экрана.
- **Иллюзия скорости от AI.** AI ускоряет код, НЕ ускоряет проверку норм — она идёт со скоростью реальных строек. Не путать «много закоммичено» с «норма доказана». Бэклог слайсов не убегает вперёд потока реальных объектов.
- **Ожидание «идеальной» нормы до запуска.** Не ждём точности — ждём диапазона + честного confidence (гл.07) и начинаем сверять. Норма уточняется данными, а не размышлением.
- **Загрязнение метрики фасовкой.** Купил мешок 25 кг, ушло 18 — это не промах нормы. Защита: сверка на расчётном количестве, `packaging_delta` отделён от `norm_delta` (11.4). Без этого reconciliation систематически шумит и ложно триггерит калибровку.
- **Голод по объектам (N=1).** Один прораб = медленный поток. Митигация: прошлые закупки для M0-prep; каждая (материал × комната) — отдельная точка данных; опционально второй прораб как источник данных (11.5). Цель — N точек на материал, а не «целый объект».
- **Дрейф scope в deferred.** Любая фича из 11.6, всплывшая в пилоте, — записать и отложить, не строить. Защита фокуса на Risk #1.
