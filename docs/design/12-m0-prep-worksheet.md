# 12 · M0-prep: бумажная валидация норм

> Рабочий пакет для слайса -1 из гл. 11. Цель: до дальнейшей полировки UI и
> платформы проверить, попадают ли seed-нормы гл. 07 в реальные закупки Давида.
> Это не продуктовый экран и не код. Это полевая таблица, которая должна дать
> первый сигнал по Risk #1.

## 12.1 Что считается готовым

M0-prep закрыт, когда по 1-2 прошлым или текущим объектам Давида есть:

- замеры комнат/поверхностей, достаточные для ручного расчёта площади;
- состав работ по ключевым материалам: плитка, клей, затирка, гидроизоляция,
  краска/грунт;
- оценка Давида до расчёта (`david_gut`) или, для прошлой стройки,
  ретроспективная оценка с пометкой `retro_gut`;
- фактическая закупка/расход: чеки, фото упаковок, остатки, возвраты или устное
  подтверждение Давида;
- таблица `estimate -> actual -> reconciliation` по каждой точке данных
  `(материал × поверхность/комната)`;
- явный verdict по каждой норме: `seed_ok`, `seed_high`, `seed_low`,
  `insufficient_data`, `excluded`.

Нужен не «идеальный датасет», а честный первый срез. Неизвестные поля лучше
оставить пустыми и понизить confidence, чем додумать их задним числом.

## 12.2 Минимум данных у Давида

### Объект

| Поле | Зачем |
|---|---|
| `project_name` | Связать расчёт, чеки и обсуждение. |
| `date_range` | Отделить старые цены/материалы от текущих. |
| `room_name` | Единица ввода и сверки. |
| `room_type` | Проверить шаблонные ожидания: bathroom/kitchen/bedroom/living/hallway/other. |
| `work_stage` | Группировка закупки: waterproofing, tiling, wall-ceiling-finish и т.д. |

### Геометрия

| Поле | Минимум | Примечание |
|---|---:|---|
| `length_m` | да | Если комната не прямоугольная, разбить на прямоугольники. |
| `width_m` | да | Для пола/потолка. |
| `height_m` | да | Для стен и гидроизоляции по стенам. |
| `openings` | желательно | Двери/окна: ширина × высота × количество. Если нет данных, считать стену gross и пометить `geometry_low_confidence`. |
| `wet_zone_height_m` | для гидро | Высота захода гидроизоляции на стены. Если неизвестна, записать фактическое правило мастера. |

### Материалы и условия

| Материал | Спросить обязательно | Почему |
|---|---|---|
| Плитка | формат A×B, толщина C, паттерн, коробка м², фактический waste/остатки | Драйвит плитку, затирку, выбор зуба, waste%. |
| Клей | бренд/SKU, кг в мешке, зуб гладилки, одинарное/двойное нанесение | Главный hidden input для kg/m². |
| Затирка | ширина шва D, бренд/SKU, кг в упаковке | Расход меняется кратно от ширины шва и формата плитки. |
| Гидроизоляция | бренд/SKU, кг в ведре/мешке, число слоёв, площадь захода | Площадь не равна только полу. |
| Краска/грунт | бренд/SKU, литры в ведре, число слоёв, основание | Впитываемость и укрывистость решают больше, чем площадь. |

### Факт закупки

| Поле | Пример |
|---|---|
| `purchased_packages` | 4 мешка клея × 20 кг |
| `returned_packages` | 1 мешок вернули |
| `leftover_amount` | осталось примерно 6 кг / половина ведра |
| `actual_used` | purchased - returned - leftover |
| `receipt_or_photo` | чек, фото упаковки, скрин заказа, голосовое подтверждение |
| `actual_confidence` | exact / estimated / rough |

Для исторических строек нормален `actual_confidence=estimated`. Такой факт можно
использовать как ранний сигнал, но не как финальный M3-тренд.

## 12.3 Таблица комнаты

Заполняется один раз на комнату.

| project | room | type | L | W | H | openings | wet | works | notes |
|---|---|---|---:|---:|---:|---|---|---|---|
| David Apt A | Bathroom | bathroom | 2.00 | 1.70 | 2.60 | door 0.70×2.00 | yes | floor tile; wall tile; ceiling paint; waterproofing | стены gross или net? |

Расчётные площади:

```text
floor_area_m2 = L * W
ceiling_area_m2 = L * W
wall_gross_m2 = 2 * (L + W) * H
opening_area_m2 = sum(width * height * count)
wall_net_m2 = wall_gross_m2 - opening_area_m2
waterproofing_area_m2 = floor_area_m2 + wet_perimeter_m * wet_zone_height_m
```

Если `wet_perimeter_m` неизвестен, считать по фактической зоне работ, а не по
красивому предположению. Для ванной это часто весь периметр, но это должен
подтвердить Давид.

## 12.4 Таблица точки данных

Одна строка = один материал на одной поверхности/комнате.
CSV-шаблон: [templates/m0-prep-points.csv](templates/m0-prep-points.csv).

| field | meaning |
|---|---|
| `point_id` | Стабильный ключ, например `aptA-bathroom-floor-adhesive`. |
| `material` | tile / adhesive / grout / waterproofing / paint / primer. |
| `surface` | floor / walls / ceiling / wet_zone. |
| `area_m2` | Площадь, на которую применяется норма. |
| `seed_formula` | Формула из гл. 07. |
| `assumptions` | Что неизвестно и взято как допущение. |
| `estimate_low` | Нижняя граница до округления фасовки. |
| `estimate_expected` | Центральная оценка до округления фасовки. |
| `estimate_high` | Верхняя граница до округления фасовки. |
| `pack_size` | Например 20 kg, 5 L, 1.44 m²/box. |
| `buy_packages` | Сколько упаковок купил бы лист. |
| `david_gut` | Оценка Давида до расчёта. |
| `actual_used` | Факт расхода, пересчитанный обратно в kg/L/m². |
| `actual_confidence` | exact / estimated / rough. |
| `norm_delta_pct` | `(actual_used - estimate_expected) / estimate_expected`. |
| `packaging_delta` | Разница от округления до упаковок; не путать с ошибкой нормы. |
| `reconciliation` | cutting / waste / buffer / reorder / one_off / unknown. |
| `verdict` | seed_ok / seed_high / seed_low / insufficient_data / excluded. |

## 12.5 Быстрые seed-формулы для M0-prep

Это не заменяет гл. 07. Это короткая шпаргалка, чтобы таблица была заполняемой за
встречу, а не превращалась в research-проект.

| Материал | Expected | Low / high | Формула M0-prep |
|---|---:|---:|---|
| Плитка | area × 1.10 | pattern-dependent | `area_m2 × (1 + waste%)`; straight=10%, diagonal=15%, herringbone=18%. |
| Клей | 3.5 kg/m² | 2-8 kg/m² | `tile_area_m2 × kg_per_m2`; если известен зуб, пересчитать через гл. 07. |
| Затирка | 0.3 kg/m² | 0.04-2.9 kg/m² | Если известны A/B/C/D: `((A+B)/(A×B))×C×D×k×area`, k=1.6 для Mapei-like seed. Иначе 0.3 kg/m² только как rough. |
| Гидроизоляция | 3.5 kg/m² | 3-4.2 kg/m² | `waterproofing_area_m2 × kg_per_m2`, минимум 2 слоя. |
| Краска | 0.17 L/m² | по rendimento | `paint_area_m2 × coats / rendimento`; seed 2 слоя при 12 m²/L. |
| Грунт | 0.10 L/m² | по rendimento | `primer_area_m2 / 10`. |

Правило: если реальный бренд/SKU известен, берём его fiche technique вместо seed.
Seed нужен только там, где M0-prep проверяет саму гипотезу до выбора точного SKU.

## 12.6 Pass/fail правила

### По строке

- `seed_ok`: `actual_used` внутри `estimate_low..estimate_high` и seed не хуже
  `david_gut`.
- `seed_high`: факт ниже диапазона или expected завышает факт сильнее, чем
  `david_gut`.
- `seed_low`: факт выше диапазона или expected занижает факт сильнее, чем
  `david_gut`.
- `insufficient_data`: факт есть, но rough/неполный настолько, что направление
  вывода ненадёжно.
- `excluded`: геометрия или факт слишком грязные; строка сохраняется как заметка,
  но не участвует в калибровке.

### По материалу

Первый M0-prep не обязан «победить». Он обязан честно сказать, что делать дальше:

- `confirm_seed`: 2+ usable точки попали в диапазон, направление ошибки не видно.
- `narrow_assumption`: строки в диапазоне, но ширина слишком большая из-за одного
  скрытого входа (например, зуб/шов/upstand).
- `shift_seed_candidate`: 2+ usable точки одного знака вне диапазона.
- `need_more_data`: меньше 2 usable точек или все факты low-confidence.

Жёсткое правило из гл. 11 сохраняется: ≥3 наблюдения одного знака вне диапазона
по материалу → норму двигать. M0-prep может только поднять кандидата на сдвиг,
не переписать NormSet автоматически.

## 12.7 Скрипт разговора с Давидом

Цель разговора — не «продать приложение», а восстановить данные по одной реальной
закупке.

1. Выбрать одну комнату, где Давид хорошо помнит закупку.
2. Зафиксировать размеры и состав работ. Если данных нет, попросить фото/план/замер,
   а не спорить по памяти.
3. Для каждого материала спросить: «Сколько ты бы заложил на глаз до расчёта?»
4. Потом поднять чек/фото/остатки и восстановить `actual_used`.
5. Отдельно спросить причину расхождения: подрезка, бой, запас, дозаказ, разовый
   выброс, неизвестно.
6. В конце показать только verdict по материалам, не весь spreadsheet.

Фраза для начала:

> «Мне не нужно, чтобы всё было идеально. Нужно понять, где нормы Obrato уже
> попадают в твою реальность, а где врут. Давай возьмём одну ванную/кухню, которую
> ты хорошо помнишь, и восстановим замеры, покупки и остатки по 5 материалам».

## 12.8 Что не делать

- Не смешивать `купили упаковок` с `реально ушло`: округление фасовки — отдельный
  `packaging_delta`.
- Не считать широкий диапазон победой. Диапазон шире ±50% не даёт доверия.
- Не додумывать зуб, шов, upstand и остатки задним числом без пометки confidence.
- Не переходить к Canvas/offline до первых usable строк `estimate -> actual`.
- Не менять NormSet по одному шумному объекту.

## 12.9 Следующий артефакт после M0-prep

Если M0-prep даёт usable строки, следующий шаг — не UI, а маленький headless flow:

```text
room input JSON/CSV -> compute-core -> purchase list CSV/PDF -> fact CSV -> delta report
```

То есть слайсы 0-2 должны замкнуть один и тот же контур данных. UI и персонаж
остаются поверх этого контура, а не заменяют его.
