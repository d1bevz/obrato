# Obrato · compute-core

Детерминированный движок прораба. **Та половина приложения, от которой зависит,
работает продукт или нет.** UI её только отображает.

```
модель ввода → нормы расхода → количества → округление до упаковок → лист закупок по этапам
(размеры + работы)  (число+допущения)              (с коэф. отхода)         (+ SVG план пола)
```

## Архитектурный инвариант

`compute-core` — **отдельный чистый модуль**: чистые функции, ноль зависимостей
от UI/DOM/браузера. Один и тот же модуль крутится:
- в **PWA** — клиентский расчёт (можно в Web Worker, вне UI-потока);
- **headless** — генерация PDF-листа для пилота (concierge).

Зачем так: когда лист закупок соврёт в количествах (а он соврёт — см. ниже), баг
пинится в юнит-тестах движка за минуту, а не ищется по всему быстро-сгенерённому
аппу. Держи эту границу чистой даже когда строишь UI рядом.

## Запуск

Нужен **Node ≥ 22.18** (нативный стриппинг TypeScript — тесты бегут без установки
зависимостей):

```bash
node --test          # юнит-тесты (5 шт., зелёные)
npm install          # только для полного typecheck
npm run typecheck    # tsc --noEmit
```

## Структура

| Файл | Что |
|------|-----|
| `src/types.ts` | Доменная модель — центр всего (Room, MaterialRequirement, Sku, PurchaseList, STAGES) |
| `src/norms/norms-table.ts` | ⚠️ Таблица норм — **невалидированные заглушки** (число + допущения) |
| `src/norms/compute-quantities.ts` | Состав работ → требования по материалам (площади, отход) |
| `src/packaging/round-to-packages.ts` | Сырое кол-во → целые упаковки (вверх) |
| `src/purchase/assemble-list.ts` | Требования → лист закупок по этапам |
| `src/geometry/floor-plan.ts` | План пола → SVG (тривиально) |
| `src/geometry/tile-layout.ts` | Раскладка плитки — **наивная сетка, ФАЗА 2 = solver подрезки** |
| `src/electric/point-templates.ts` | Электроточки по шаблону комнаты (= общий паттерн ввода) |
| `src/catalog/sample-catalog.ts` | ⚠️ SKU-каталог — заглушка; на пилоте курируется вручную |
| `src/pipeline.ts` | `buildProjectOutput()` — главный вход |

## ⚠️ Главное: нормы — это Risk #1, и они НЕ валидированы

Коэффициенты в `norms-table.ts` — стартовые заглушки. Часть реальных входов (зубец
шпателя, паттерн раскладки, ровность основания под стяжку) в модели ввода Д×Ш×В
**отсутствует** — они зашиты как допущения и едут в каждый результат
(`assumptions.notes`), чтобы при сверке было видно ГДЕ движок соврал.

**Задача первого пилота — не угадать эти числа здесь, а узнать настоящие**: прогнать
один реальный объект Давида через движок → PDF-лист → сверить каждое количество с его
фактической закупкой. См. `test/pipeline.test.ts` (TODO в конце) и `docs/design.md`
(Risk #1, The Assignment).

## Что сознательно НЕ здесь (фаза 2)

Solver балансировки подрезки плитки, разводка электрики трассами, непрямоугольная
геометрия, отслеживание отклонений по ходу стройки (drift), Bluetooth-дальномер.
